### 1.2 Безопасность

| Проблема | Серьёзность | Файл:строка | Описание |
|----------|-------------|-------------|----------|
| Утечка ошибок в UI | Средняя | `admin.py:186`, `admin.py:224` | `f"Ошибка: {e}"` — пользователю показывается сырой traceback. Может раскрыть внутреннюю структуру БД |
| Bare except | Низкая | `main.py:111` | `except: pass` — глушит все исключения без логирования |
| Отсутствие валидации spreadsheet_id | Средняя | `family.py:147` | `ss_id = url.split("/d/")[1].split("/")[0]` — нет валидации формата ID (допустимы только `[a-zA-Z0-9_-]`). Потенциально вредоносный spreadsheet_id может попасть в Google API запрос |
| In-memory состояния | Средняя | `communication.py:14-16` | `user_states` и `admin_msg_map` — словари в памяти. При перезапуске бота все состояния теряются. Если пользователь в процессе рассылки — данные пропадут |
| Прямой SQL без get_db_connection | Низкая | `admin.py:18` | `is_user_admin()` использует `sqlite3.connect(DB_PATH)` напрямую, минуя context manager с WAL и timeout |
| Нет rate limiting для пользователей | Средняя | Везде | Отсутствует ограничение частоты запросов от пользователей. Злоумышленник может спамить `/grades` и вызвать лавину запросов к Google Sheets API |
| Хранение телефонов | Низкая | `communication.py:71` | Телефон выводится в admin group card — потенциальная утечка PII при компрометации группы |

### 1.3 Масштабируемость

| Узкое место | Влияние | Описание |
|-------------|---------|----------|
| SQLite | Высокое | При >100 семей одновременные записи в БД будут блокироваться (single-writer). WAL помогает, но не решает проблему полностью |
| Polling Google Sheets | Высокое | Каждый студент = 2 API-вызова за цикл (`get_spreadsheet_title` + `get_sheet_data`). При 50 студентах = 100 запросов каждые 5 минут. Квота Google Sheets API = 300 запросов/мин — потолок ~150 студентов |
| `get_spreadsheet_title` в каждом цикле | Высокое | `monitor_engine.py:54` — заголовок таблицы запрашивается при КАЖДОМ цикле проверки, хотя он практически не меняется. Это удваивает количество API-запросов |
| Рассылка в основном потоке | Среднее | `communication.py:209-220` — broadcast блокирует обработку других сообщений. При 1000 пользователей: ~50 секунд простоя |
| Создание сервиса Sheets API | Среднее | `google_sheets.py:33` — `get_sheets_service()` вызывается при каждом запросе. Сервис не кэшируется — повторная авторизация и HTTP-сессия каждый раз |
| Одинарный поток мониторинга | Среднее | Все студенты проверяются последовательно в одном потоке. Задержка на одном студенте задерживает уведомления для всех остальных |

### 1.4 Взаимодействие с API Telegram

| Аспект | Статус | Рекомендация |
|--------|--------|--------------|
| Polling vs Webhook | Polling (`bot.polling(none_stop=True)`) | Для production рекомендуется Webhook — меньше нагрузка, мгновенная доставка. Особенно важно при деплое на VPS |
| Обработка Flood Control | Частичная | Broadcast имеет `time.sleep(0.05)`, но нет обработки `RetryAfter` исключений от Telegram |
| Cleanup старых сообщений | Хорошо | `send_menu_safe` удаляет предыдущее навигационное сообщение — чат остаётся чистым |
| Callback query ответы | Частично | Не все callback handlers вызывают `answer_callback_query()` — пользователь видит "часики" |

## 2. Аудит бизнес-логики

### 2.1 User Journey: /start до целевого действия

```
/start
  └─ Новый пользователь → Запрос контакта → Проверка в БД
       ├─ Найден + привязан к семье → Главное меню (оценки приходят автоматически)
       ├─ Найден, но НЕ в семье → "Ожидайте привязки" (ТУПИК)
       └─ Не найден → "Свяжитесь с админом" (ТУПИК)
```

**Проблема:** Два тупиковых сценария из трёх не предлагают конкретного действия. Пользователь не знает, как связаться с администратором.

### 2.2 Тупиковые сценарии

| Сценарий | Файл:строка | Проблема | Решение |
|----------|-------------|----------|---------|
| Незарегистрированный пользователь | `main.py:93-98` | Сообщение "свяжитесь с администратором" без контакта/ссылки | Добавить inline-кнопку с ссылкой на админа или форму заявки |
| Пользователь без семьи | `main.py:46-48` | "Ожидайте привязки" — нет индикации прогресса | Добавить кнопку "Напомнить администратору" |
| Ошибка Google Sheets при добавлении ребёнка | `family.py:166` | Если API недоступен, ребёнок создаётся как "Новый ученик" | Спросить имя вручную или отложить добавление |
| Глава семьи без детей | UI | Есть кнопка "Оценки", но она ничего не покажет | Показывать подсказку "Сначала добавьте ребёнка" |
| Истекшая подписка | DB schema | Поле `subscription_end` существует, но нигде не проверяется | Реализовать проверку подписки или удалить поле |

### 2.3 Логичность цепочек сообщений

| Цепочка | Оценка | Замечание |
|---------|--------|-----------|
| Создание семьи | 4/5 | Логичный flow: имя → глава → результат. Но нет промежуточного подтверждения |
| Добавление ребёнка | 3/5 | Нет проверки, что таблица содержит лист "Сегодня". Пользователь узнает о проблеме только когда оценки не придут |
| Добавление родственника | 3/5 | Парсинг "ФИО Телефон" через `split(maxsplit=2)` хрупкий — если ФИО из 3 слов, телефон не распарсится |
| Поддержка | 4/5 | Хороший flow с карточкой пользователя для админа. Но admin_msg_map теряется при перезапуске |
| Рассылка | 4/5 | Есть подтверждение перед отправкой. Но нет preview для админа и нет возможности отмены в процессе |

### 2.4 Возможности для автоматизации

1. **Автоочистка grade_history** — записи "Сегодня" накапливаются бесконечно. Нужен cron для архивации по дням
2. **Автоуведомление о неактивности** — если ребёнок не получал оценок >3 дней (выходные, каникулы), уведомить родителей о статусе
3. **Автопривязка родственника** — когда глава добавляет родственника, а тот потом авторизуется, привязка происходит автоматически (это уже работает через телефон)
4. **Ежедневная сводка** — вечерний дайджест всех оценок за день вместо отдельных уведомлений

## 4. Найденные баги

| # | Серьёзность | Файл:строка | Описание |
|---|-------------|-------------|----------|
| 1 | Высокая | `data_cleaner.py:48` | Тест `("болел", (None, "болел"))` **падает** — "болел" не входит в whitelist `['н', 'б', 'н/а', 'осв']`, функция вернёт `(None, None)` вместо `(None, "болел")` |
| 2 | Высокая | `data_cleaner.py:49` | Тест `("5.0", (5.0, "5.0"))` **падает** — regex `^[1-5][\-+=]?$` не матчит "5.0", а "5.0" не в whitelist. Результат: `(None, None)` |
| 3 | Средняя | `database_manager.py:447` | Дублированный `return stats` — unreachable code |
| 4 | Средняя | `admin.py:112` | `f_id = int(call.data.split('_')[2])` — callback_data `admin_manage_5` разбивается как `['admin', 'manage', '5']`, это работает. Но `confirm_delete_family_5` (строка 129) разбивается как `['confirm', 'delete', 'family', '5']` и берётся `[3]` — корректно, но формат парсинга непоследователен и хрупок |
| 5 | Низкая | `main.py:39` | `send_menu_safe` с Markdown `*Супер-администратор*`, но `parse_mode='HTML'` в `send_menu_safe` — `*` не будут работать как bold |
| 6 | Низкая | `family.py:76` | Для членов семьи показывается глобальная `role` (всегда "senior"), а не контекстная роль в семье. Глава видит своих членов как "(senior)" |

### 5.2 Инновационные идеи

#### Идея 1: Telegram Mini App — Дашборд оценок
Интерактивная веб-страница внутри Telegram с графиками динамики оценок, средним баллом по предметам, сравнением с прошлой неделей. Визуализация через Chart.js. Родителям не нужно открывать Google Sheets — всё внутри бота.

#### Идея 2: AI-анализ успеваемости
Подключить Claude API для анализа трендов: "У вашего ребёнка снижается математика последние 2 недели — средний балл упал с 4.5 до 3.8". Еженедельный автоматический отчёт с рекомендациями.

#### Идея 3: Геймификация для учеников
Отдельный режим для самих учеников (если разрешат родители): streak-и за пятёрки, бейджи за прогресс, weekly leaderboard среди одноклассников (анонимизированный). Увеличит вовлечённость и мотивацию.

### 5.3 Рекомендации по удержанию (Retention)

| Механика | Описание | Ожидаемый эффект |
|----------|----------|------------------|
| **Ежедневная сводка (19:00)** | Вечерний дайджест: "Сегодня у Заура: Математика 5, Физика 4-. Средний балл: 4.5" | Формирует привычку проверять бота ежедневно |
| **Еженедельный отчёт (ПН)** | Сравнение с прошлой неделей, лучший/худший предмет, тренд | Даёт родителям ощущение контроля |
| **Push при отсутствии оценок** | "Сегодня оценок пока нет — возможно, был выходной или каникулы" (после 15:00) | Показывает, что бот работает даже когда "молчит" |
| **Празднование достижений** | "Заур получил 5 по математике — это уже 5-я пятёрка подряд!" | Позитивные эмоции = привязанность к боту |
| **Напоминание неактивным** | Если пользователь не открывал бота 7 дней — мягкое push-уведомление со сводкой | Возвращение "спящих" пользователей |
| **Onboarding-цепочка** | День 1: "Вот как работает бот". День 2: "Первые оценки!". День 7: "Ваша первая неделя в цифрах" | Снижает отток новых пользователей |
