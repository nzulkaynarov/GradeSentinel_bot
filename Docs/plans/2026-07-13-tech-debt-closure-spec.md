**Исполнители:** Opus-агенты, по одному на пакет. TD-1…TD-4 — параллельно (файлы не пересекаются),
TD-5 — после них (описывает итоговую структуру). Каждый пакет = ветка от `main` + локальный коммит,
БЕЗ push (push делает оркестратор). Merge — владелец.

**Общий контекст для КАЖДОГО агента (вставлять в промпт):**
- PostgreSQL 17 / psycopg v3: плейсхолдеры `%s`; `get_db_connection()` — context manager (commit на
  выходе, rollback при exception); для атомарности нескольких операций — `conn_or_new(conn=None)`
  из `src/db/pg.py`; Row — Mapping; PG отдаёт datetime/date/bool объекты.
- Бот СИНХРОННЫЙ (pyTelegramBotAPI, polling, threaded worker pool). Никаких async.
- i18n: user-строки только через `t(key, lang)`; 3 локали ru/uz/en синхронны (sync-тесты).
- Порядок регистрации message-handlers критичен: state_flows → navigation → ai_chat → остальные
  (регистрация происходит при импорте модулей в main.py:72-86).
- Тесты: `docker compose -f docker-compose.test.yml run --rm -e BOT_TOKEN='12345:ci-test-token'
  -e ADMIN_GROUP_ID='0' tests` — полный сюит ЗЕЛЁНЫЙ перед завершением (база: 507 passed на main).
- `webapp/` НЕ трогать. Прод — реальные пользователи; поведение сохранять, если пакет не говорит иначе.
- Рефакторинг-пакеты (TD-2/3/4): публичные import-пути СОХРАНЯЮТСЯ через re-export — ни один
  внешний файл не должен потребовать правки import'ов (кроме случаев, явно указанных в пакете).

## TD-1 — Атомарность уведомлений (бывший PR-F: B8, B9, B10, B11, B14 из тех-аудита)

**Ветка:** `fix/notification-atomicity` от main.
**Файлы:** `src/monitor_engine.py`, `src/schedulers.py`, `src/db/grades.py`, `src/db/notifications.py`,
`src/notifications/*`, тесты.
**Принятое дизайн-решение (не обсуждается):** БЕЗ persistent-outbox. Масштаб (десятки семей) не
оправдывает новую таблицу-очередь; вместо этого — сужение окон потери до per-student и
идемпотентность. Outbox — отдельный RFC при >100 семей.

1. **B8 (HIGH):** в `_check_for_new_grades_impl` уведомления шлются ОДНИМ проходом в конце цикла по
   всем студентам → exception в фазе send теряет ВЕСЬ батч навсегда (след. цикл видит «не изменилось»).
   Фикс: отправка batch'а данного студента сразу после успешной записи его оценок в БД (внутри
   пер-студенческого блока, в try/except — сбой отправки одного студента не прерывает цикл, логируется
   с тегом `[NOTIFY_FAIL]`). Групповые уведомления — тем же принципом. Батчинг по студенту сохранить.
2. **B11 (MEDIUM):** `update_grade_by_content` (src/db/grades.py:106-118) обновляет по
   (student, subject, date) БЕЗ raw_text/id → при мульти-строках на день ловит UNIQUE-abort.
   Фикс: `get_existing_grade_by_content` уже возвращает строку — обновлять по её `id`.
3. **B9 (HIGH):** morning-flush: `get_and_clear_queued_notifications` удаляет из очереди ДО отправки
   (commit на выходе with-блока), сбой отправки = безвозвратная потеря (групповая очередь не
   реконструируется). Фикс: выбрать id'шники → отправить → удалить ТОЛЬКО отправленные (per-message
   или per-recipient), в отдельной транзакции после send. Повторный запуск дошлёт остаток.
4. **B10 (HIGH):** evening/morning/weekly-джобы: маркер ставится после ПОЛНОГО прогона → частичный
   сбой = двойная рассылка уже получившим. Фикс: per-recipient идемпотентность — маркер
   `{job}_{date}_{telegram_id}` в settings (или множество отправленных в одном json-маркере),
   проверка перед send.
