## Контекст-константы (в промпт КАЖДОМУ агенту — не нарушать)

- **PostgreSQL 17 / psycopg v3**: плейсхолдеры `%s`; `get_db_connection()` — context manager (commit на
  выходе, rollback при exception); для атомарности нескольких операций передавать один `conn` через
  `conn_or_new(conn)` (`src/db/pg.py`); Row — Mapping (`row['c']` и `row[0]`); PG отдаёт
  datetime/date/bool/float ОБЪЕКТЫ (не строки); даты для дат используй `to_date_str()`; ташкентское
  «сейчас» = UTC+5 (есть хелперы, не изобретать).
- **Бот СИНХРОННЫЙ** (pyTelegramBotAPI polling); потоки: main + monitor + scheduler + heartbeat +
  broadcast. Порядок регистрации message-handler'ов важен (state_flows → navigation → ai_chat → rest).
- **i18n**: все user-строки через `t()`; локали ru/uz/en синхронны (есть sync-тесты — новый ключ во все три).
- **Схема — Alembic**, head = `0002_payment_hardening`. Новая миграция: `down_revision="0002_payment_hardening"`,
  вручную через `op.*` (env.py без target_metadata).
- **Тесты**: `docker compose -f docker-compose.test.yml run --rm -e BOT_TOKEN='12345:ci-test' -e ADMIN_GROUP_ID='0' tests pytest -q`.
  База — **507 тестов** (506 + weekly-fix PR #107 после merge). Полный сюит зелёный перед завершением.
- **Прод, авто-деплой push→main.** НЕ пушить/мержить/трогать main. Ветка + локальный коммит
  (в теле `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`). Вернуть diff-сводку + список тестов.
- **Инвариант рефакторинга:** публичный API модулей и импорты `from src.database_manager import ...` /
  `from src.handlers.subscription import ...` НЕ ломать (много внешних call-site'ов + тесты импортируют по
  старым путям). Обратная совместимость через re-export.

## ЧАСТЬ 1 — ТЕХДОЛГ НАДЁЖНОСТИ (Волна A, приоритет)

### PR-F1 · Атомарность уведомлений: outbox вместо write-then-notify

**Проблема (подтверждено кодом `src/monitor_engine.py:304-529`).** Оценки пишутся в `grade_history`
(`add_grade`, стр. 443) в пер-студенческом цикле, а батч уведомлений отправляется ОТДЕЛЬНЫМИ циклами в
конце (стр. 474+, 505+), после обработки всех студентов. Любой exception в фазе отправки/форматирования
ловится `report("monitor.cycle", e)` (стр. 705) → цикл завершается, оценки в БД, батч НЕ отправлен. На
следующем цикле `old_clean == new_clean` → `continue` → **уведомление не уйдёт НИКОГДА**. Окно потери —
весь цикл всех студентов. Это последний крупный незакрытый риск надёжности (инциденты ночного спама
21.05 были про дубли; этот — про тихую потерю).

**Дизайн (решение принято — реализуй его; если видишь изъян, опиши в отчёте, но не меняй молча).**
Персистентный outbox через флаг на строке оценки, БЕЗ новой сложной машинерии (масштаб — единицы семей):

1. Миграция: добавить `grade_history.notified_at timestamptz NULL` (NULL = ещё не доставлено;
   индекс частичный `WHERE notified_at IS NULL`). Аналогично для четвертных, если они шлют уведомления.
2. `add_grade` пишет строку с `notified_at = NULL`.
3. **Отправка сразу после успешной записи по КАЖДОМУ студенту** (перенести send внутрь пер-студенческого
   блока — батчинг по `(tg_id, student_id)` при этом сохраняется, т.к. ключ уже пер-студенческий). После
   подтверждённой отправки (или постановки в `notification_queue` в тихие часы — это тоже «доставлено, но
   отложено») — проставить `notified_at = now()`. Постановка в очередь тихих часов считается успешной
   доставкой (сообщение персистентно в очереди).
4. **Sweeper в начале каждого цикла**: выбрать оценки с `notified_at IS NULL` старше ~1 цикла, дослать и
   пометить. Это добивает то, что не ушло из-за краха между write и send.
5. Групповые уведомления (`_send_to_groups_for_student`) — тем же принципом; учесть, что групповая
   очередь тихих часов (`group_notification_queue`) уже персистентна.

**Взаимодействие с тихими часами:** уведомление, положенное в `notification_queue`/`group_notification_queue`,
= доставлено (persistent). Не помечай `notified_at` только для реально «повисших» (не отправленных и не
поставленных в очередь). Разбери путь `should_defer` в `src/notifications/quiet_hours.py`.

**Что НЕ трогать:** двухфазный `_pending_grades` (это анти-ghost дебаунс, отдельная тема; вердикт аудита —
не удалять). content-based identity. Формат уведомлений (HTML).

**Тесты (обязательно, новый `tests/test_monitor_outbox.py`):**
- crash между `add_grade` и send (симулируй exception в фазе отправки) → оценка в БД с `notified_at IS NULL`
  → на следующем цикле sweeper дошлёт (мок Sender, проверить факт отправки ровно один раз).
- happy-path: отправлено → `notified_at` проставлен → повторный цикл НЕ шлёт дубль.
- тихие часы: оценка → в `notification_queue`, `notified_at` проставлен, дубля нет.
- групповой путь аналогично.

### PR-F2 · Недеструктивное чтение очереди + per-recipient идемпотентность

**Проблемы (`src/schedulers.py`).** (B9) `get_and_clear_queued_notifications` (morning flush ~стр. 288)
удаляет из очереди в той же транзакции, что и SELECT, commit ДО отправки → краш отправки = безвозвратная
потеря (групповая очередь не реконструируется из grade_history). (B10) evening/morning/weekly ставят
маркер только после ПОЛНОГО прохода → частичный сбой рассылки → следующий tick в окне (`minute<6`,
sleep 180) перезапустит с нуля → часть родителей получит сводку дважды.

**Дизайн.** (B9) Читать очередь БЕЗ немедленного удаления; удалять per-message только после
подтверждённой отправки (или пометка `sent_at`, затем чистка доставленных). (B10) Ввести per-recipient
чек-пойнт: маркер вида `<job>_sent_<recipient>_<date>` в `settings` (как уже сделано для
`sub_expiry_sent_*`), проверять перед отправкой каждому.

**Тесты:** сбой send после чтения очереди не теряет сообщения; частичный сбой рассылки не даёт дублей
при повторном запуске джоба в окне.

**Зависимость:** PR-F1 и PR-F2 оба трогают `monitor_engine.py`/`schedulers.py`/`src/db/notifications.py`
→ делать ПОСЛЕДОВАТЕЛЬНО, стеком: F2 от ветки F1. Один агент на оба (одна зона), либо F2 стеком на F1.

```
Волна A (надёжность, последовательно):     Волна B (модуляризация, параллельно A):
  PR-F1 (outbox) ── от main                  PR-M1 (ai prompts)  ── от main   [первым, низкий риск]
  PR-F2 (queue/idempotency) ── стек на F1     PR-M2 (subscription pkg) ── от main
                                             PR-M3 (main panel) ── от main
```
Пересечения файлов: A трогает `monitor_engine.py`, `schedulers.py`, `src/db/notifications.py`, миграции.
B трогает `analytics_engine.py`, `subscription.py`, `main.py`. **A и B не пересекаются** → параллельно.
Внутри B: M1/M2/M3 разные файлы → параллельно (main.py у M3, subscription у M2, analytics у M1). При
merge порядок: сначала A (миграция F1), потом B в любом порядке — конфликтов не ожидается (проверить
локальным test-merge перед PR, как делали в прошлый заход).

   импортируют по ним).
