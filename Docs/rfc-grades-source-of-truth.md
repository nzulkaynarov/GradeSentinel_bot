# RFC: единый источник правды для оценок и стабильный grade_date

**Status:** draft
**Автор:** Claude (черновик 14.05.2026, ревью у владельца)
**Контекст инцидента:** анализ прод-БД 13–14.05.2026 после жалоб «бот шлёт неверные уведомления»

---

## TL;DR

Текущая архитектура опирается на `cell_reference` как stable identifier и на `date(date_added)` как фактическую дату оценки. Оба предположения нарушаются:

1. **Race с двухфазным подтверждением** (уже зафикшено в `fef55c9`) порождал дубли при одновременной работе monitor и `_maybe_sync_all_grades`.
2. **Сдвиг шапки в Sheets** — учитель добавил/удалил столбец после массового импорта 10.05 → у 830 из 846 записей `date_added` теперь смещён на 1 день относительно реальной шапки. cell_reference из БД больше не указывает на ту дату что в момент импорта.
3. **Битый парсер «мая → март»** (зафикшен `7f592ce`) оставил 16 записей student=1 с датами на ~2 месяца раньше реальной.

Симптом для родителя — недостоверные даты в `/grades`/дашборде/AI и «лишние» уведомления при race.

Предлагается:
- Ввести колонку `grade_date NOT NULL` в `grade_history`. Дата оценки хранится явно, не вычисляется через `date(date_added)`.
- Новый UNIQUE по `(student_id, subject, grade_date, raw_text)` вместо `(student_id, cell_reference)`. `cell_reference` остаётся как debug-info.
- Унифицировать источник: monitor читает только «Все оценки». Листы «Сегодня» / «Неделя» — view для родителей, бот их не парсит.
- Двухфазное подтверждение убрать (race исчезает по дизайну при едином источнике).

## Текущая архитектура — что есть и где ломается

```
                 каждые 5 мин
monitor_engine ───────────────► Sheets "Сегодня"  ──► add_grade(cell_ref="Сегодня!...")
       │                                                │
       │   в том же main-loop, раз в час                 │
       └─► _maybe_sync_all_grades ─► Sheets "Все оценки"
                                  ─► Sheets "Неделя"     │
                                                          │
                                                          ▼
                                                  grade_history
                                                  UNIQUE(student, cell_reference)
                                                  date_added = SQL CURRENT_TIMESTAMP
                                                              ИЛИ artificial "YYYY-MM-DD 12:00:00"
```

### Проблема 1: race monitor pending ↔ history-sync

См. `fef55c9` (PR #29). Зафикшено: history-sync пропускает записи с today (Tashkent).

### Проблема 2: cell_reference нестабилен против мутаций шапки

`_col_letter(col_index)` сохраняет в `cell_reference` физическую позицию ячейки на момент импорта. Если учитель потом вставит/удалит колонку — все cell_reference в БД указывают на сдвинутые ячейки. При следующем парсинге дедуп уже не находит совпадения (или, что хуже, отдаёт ложную дату).

Проявление в проде: 830 записей с `date_added` сдвинутым на 1 день относительно реальной шапки (предположительно после удаления столбца «1 сентября» учителем).

### Проблема 3: date_added одновременно как timestamp и как «дата оценки»

В `_import_from_sheet` `date_added = rec['date'].strftime('%Y-%m-%d 12:00:00')` — это **дата оценки** замаскированная под технический timestamp.

В `monitor.add_grade` `date_added = CURRENT_TIMESTAMP` — это **момент INSERT'а**, отличается от даты оценки в UTC vs Tashkent на ±1 день возле полуночи.

В `update_grade` `date_added = CURRENT_TIMESTAMP` — `date_added` записи **меняется при каждом UPDATE**. Дата оценки утекает.

Дедуп через `date(date_added)` ломается во всех трёх случаях.

### Проблема 4: два источника + опечатка-защита через ожидание цикла

Двухфазное подтверждение возникло как защита «учитель опечатался → стёр». Если перейдём на единый источник «Все оценки» — учителя там почти не стирают (мастер-лист). Защита излишняя.

## Предлагаемая архитектура

### Schema

```sql
CREATE TABLE grade_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    student_id INTEGER NOT NULL REFERENCES students(id),
    subject TEXT NOT NULL,
    grade_value REAL,
    raw_text TEXT NOT NULL,
    grade_date DATE NOT NULL,            -- NEW: фактическая дата оценки
    cell_reference TEXT,                 -- debug-only, NULLABLE
    date_added TIMESTAMP DEFAULT CURRENT_TIMESTAMP,  -- технический
    UNIQUE(student_id, subject, grade_date, raw_text)
);
CREATE INDEX idx_gh_student_date ON grade_history(student_id, grade_date);
```

`UNIQUE` через `(subject, grade_date, raw_text)` — стабилен к любым мутациям шапки.

### Поток данных

```
            каждые 5 мин
monitor ──► Sheets "Все оценки" → парс шапки → для каждой непустой ячейки:
                                              записать (subject, grade_date, raw_text)
                                              где grade_date пришёл из header колонки

                            если grade_date == today И запись новая
                            (для (student, subject, grade_date)) → notify
                            (иначе silent backfill / silent update)
```

«Сегодня» и «Неделя» бот **не читает**. Двухфазное подтверждение **удаляется**.

### Что упрощается

- Один цикл, один источник, нет race.
- Дедуп по содержимому — атомарный на уровне БД через UNIQUE.
- При сдвиге шапки старые записи **сами по себе** остаются с корректным `grade_date`. Новый импорт находит их через UNIQUE → SKIP.
- `update_grade` либо вообще не нужен (просто INSERT-or-replace), либо обновляет только `raw_text`/`grade_value`, не дату.
- `monitor_engine` тоньше: только парсинг + диф + уведомление.

### Что мы теряем

- **Real-time для оценок которых ещё нет в «Все оценки», но есть в «Сегодня».** Если учитель пишет сначала в «Сегодня» и потом переносит в «Все оценки» с задержкой — уведомление придёт после этого переноса.
  - Требует проверки у класса: какая реальная задержка? Если синхронно (через formula/auto-fill) — потерь нет.
  - Если задержка есть — рассмотреть гибрид: читать ОБА листа, но писать в одну запись по (subject, grade_date), не по cell_reference.

## Этапы (предлагаемое разбиение на PR'ы)

### Этап 0 — аудит и точечная чистка (1 день, **уже не делаем сейчас**)

Решено: пропускаем тактическую чистку, идём к рефакторингу. Симптом «дата на 1 день меньше» останется в `/grades` до выкатки этапа 2.

### Этап 1 — schema migration: добавить `grade_date`

PR: `feat(schema): добавить grade_date в grade_history с бэкфиллом`.

- Миграция `init_db()`: `ALTER TABLE grade_history ADD COLUMN grade_date DATE`. Сначала без NOT NULL.
- Backfill skript: для каждой записи восстановить `grade_date`:
  - запись с `cell_reference LIKE 'Сегодня!...'` → парсим дату из cell_ref (`Сегодня!{subject}:{date}` имеет дату внутри);
  - запись с `cell_reference LIKE 'Все оценки!...'` или `Неделя!...` → читаем header текущей шапки Sheets по col_letter. **Внимание:** если шапка сдвинулась после импорта, эта дата всё ещё актуальна на момент бэкфилла. Зафиксируем её — дальнейшие сдвиги шапки не сломают БД.
  - запись без cell_reference (если такие появятся) → fallback на `date(date_added)`.
- После бэкфилла: `UPDATE grade_history SET grade_date = ... WHERE id IN (...)`.
- Финальная миграция: `NOT NULL` + новый UNIQUE.
- Старый UNIQUE по cell_reference — отбросить.

Тесты:
- Регресс на текущей структуре листов (xlsx ученика как fixture).
- Бэкфилл-функция unit-tested с моком Sheets.

### Этап 2 — `/grades`, дашборд, AI читают `grade_date`

PR: `refactor(read-path): использовать grade_date вместо date(date_added)`.

- `database_manager.get_grade_history_for_student` и аналоги — переключаются на `grade_date`.
- WebApp: `compute_summary`, `compute_trend_by_day`, `compute_by_subject` — читают `grade_date`.
- AI-аналитика — то же.

Тесты:
- `test_webapp_dashboard.py` обновить на `grade_date`.
- Regression на «бот показывает правильную дату».

### Этап 3 — `monitor_engine` и `_import_from_sheet` пишут `grade_date`

PR: `feat(write-path): писать grade_date явно, дедуп через UNIQUE по содержимому`.

- `add_grade(student, subject, grade_date, raw_text, value, cell_ref=None)` — обновлённая сигнатура.
- `_import_from_sheet`: формирует `grade_date` из header, вставляет в новую колонку.
- `monitor_engine`: формирует `grade_date` из cell_reference «Сегодня!subject:date».
- `IntegrityError` на UNIQUE — нормальный путь дедупа (как было с cell_reference).

### Этап 4 — единый источник: убрать чтение «Сегодня» и «Неделя»

PR: `refactor(source): монитор читает только «Все оценки»`.

Под фича-флагом `MONOSOURCE_GRADES=1` (env-flag). По умолчанию — старый путь, но фикс grade_date уже работает. После недели тестирования с флагом — флаг становится дефолтом.

Условие приёмки: за тестовую неделю количество уведомлений в новом режиме совпадает с тем что было бы у старого + явно объяснимые отклонения.

### Этап 5 — убрать двухфазное подтверждение

PR: `refactor(monitor): убрать pending после перехода на единый источник`.

Только после успешного этапа 4. Удаляем `_pending_grades` со всеми хелперами и тестами `tests/test_monitor_pending.py`.

### Этап 6 (опционально) — четвертные оценки

Quarter_grades сейчас работает чисто (UNIQUE + UPSERT). Стоит ли добавлять двухфазное подтверждение для защиты от опечаток в годовой/четвертной — обсуждаемо. Считаю — да, по той же причине что и для текущих, но риск опечаток у учителей в финальных оценках обычно ниже.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Бэкфилл grade_date по сдвинутой шапке — кодифицирует сдвиг | Документируем как «текущее лучшее знание»; при будущих сдвигах данные уже не пострадают потому что cell_reference больше не используется |
| UNIQUE по (subject, grade_date, raw_text) теряет легитимные кейсы «две одинаковые оценки за один день один предмет» | Редкий кейс. Логируем `IntegrityError` через WARN, не FAIL. При жалобах — расширяем UNIQUE добавлением `cell_reference`. |
| Этап 4 ломает уведомления массово | Фича-флаг + shadow mode 24ч + сравнение объёма уведомлений. |
| Учитель делает крупную правку в момент миграции | Этап 1 атомарен (одна транзакция). Этап 4 — отдельный PR, флаг можно мгновенно вернуть в OFF. |
| Sheets quota на 2x чтении в этапе 4 (Все оценки больше Сегодня) | Запас по квоте 15x. Не критично. |
